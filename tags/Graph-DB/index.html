LSched: A Workload-Aware Learned Query Scheduler for Analytical Database Systems

摘要

  • 查询调度是分析数据库系统的一项关键任务，它会极大地影响查询延迟。
  • 最近的一项试验提出使用强化学习来自动学习端到端调度策略。然而，此类试验无法考虑数据库特定的特征（例如，操作员类型、流水线），因此对于分析数据库系统来说效率不高。
  • 在本文中，我们试图填补这一空白，并介绍了 LSched（学习调度程序），一种用于内存分析数据库系统的完全学习的工作负载感知查询调度程序。
  • LSched 为动态分析工作负载提供了高效的查询间和查询内调度（即，不同的查询可以随时到达/离开）。

介绍

  • 分析查询处理系统中的任务调度可以对查询延迟和查询吞吐量产生深远的影响。

  • 现代数据库系统中，一个查询通常是由一个或多个子任务组成的。查询调度程序负责从所有当前运行的查询中找到所有任务的执行顺序，以优化某些目标。然后，优化目标确定了资源（例如CPU、内存、线程）如何在并发查询之间共享，以及系统的各个部分如何协同工作。

  • 然而，对于最常见的优化目标（例如，总延迟），导出最优算法是不可能的或在计算上不切实际的。

  • 因此，常用的调度算法依赖于特定的启发式，包括FIFO、公平调度、短作业优先（SJF）、最高优先级优先（HPF）、打包和近似算法。

  • 尽管这些标准技术易于实现且透明（即，决策是根据指导原则做出的，因此是可以理解的），但在调度分析工作负载时，它们显然错过了主要的性能优化机会。

  • 启发式倾向于简化假设，众所周知，很难将其扩展到考虑更复杂的系统行为。

  • Decima 提出使用强化学习为输入工作负载自动构建特定的端到端调度策略，但无法考虑数据库特定的特性，例如查询结构、不同的运算符（例如聚合与连接）以及流水线机会。例如，Decima 假设任何查询都是任务的DAG，其中每个任务都是一个黑匣子，除非其所有父任务都已完成（即不支持流水线），否则无法对其进行调度。然而，在关系查询的情况下，至关重要的是将每个任务视为一个白盒，并根据其细节以及与其他任务的关系优化调度策略。

  • 典型关键路径流水线、Decima、与学习调度器对比

  • 利用机器学习分析工作负载中的调度查询，此类试验倾向于使用ML来预测查询或任务延迟，以做出更好的决策，而不是试图学习整个调度策略。

  • Quickstep 使用线性回归根据给定查询的执行历史预测未来工单的执行时间。这种预测用于控制来自运行启发式调度策略（例如 HPF）的资源分配决策。

  • SelfTune 采用了约束优化技术来调整其固定调度策略的超参数，以适应输入工作负载。

  • 另一项工作使用 RL 自动学习执行不同查询的顺序，以最大限度地利用缓冲池项（即提高命中率）。

  • 然而，仅根据缓冲池命中数优化调度，而不考虑其他调度因素（例如，每个查询的并行度）和执行环境的当前状态，并不一定会产生更好的多查询执行计划。

  • 在本文中，我们介绍了 LSched（学习调度器），一种用于内存分析数据库系统的基于完全 RL 的学习工作负载感知查询调度器。

  • LSched 为动态分析工作负载提供了高效的查询间和查询内调度（即，不同的查询可以随时到达和离开）。

  • 给定过去执行的一组查询计划（例如工作负载日志），LSched 自动学习如何在每个调度事件中做出以下决策：（1）要执行的查询子集，（2）要执行的查询中的运算符子集，以及（3）要分配给这些运算符的资源量（例如线程）。

  • LSched 根据用户定义的高级策略目标（例如，最小化用户感知的延迟），针对特定输入工作负载优化这些调度决策。

  • LSched 的工作原理如下：它将需要调度的查询作为输入（查询可以批量或流式到达），并使用典型的分析 DBMS（例如 Quickstep）为其生成查询计划（即算子的 DAG）。然后，将该查询计划馈送给 LSched 的调度代理，该调度代理由查询编码器和调度预测器组成，以预测用于执行该查询的调度决策序列以及当前系统中已经存在的其他查询。每个调度决策都由调度事件触发（例如，新查询的到达或运算符的执行完成），并设计为在运行整个调度决策序列（即长期执行计划）后最小化所有现有查询的预期延迟。随着时间的推移，LSched 发现更好的调度决策，LSched 的调度预测器会改进并产生更好的执行计划。此改进过程将继续，直到预测值收敛。

  • 总之，LSched 有以下贡献，克服了 Decima 的局限性，并概述了如何在数据库上下文中学习端到端查询调度：

    • 利用高效的物理计划特性：非黑盒
    • 高效准确的查询编码：自定义树卷积 + GAT
    • 灵活的调度策略：自动控制操作符流水线
    • 平均和尾部延迟之间的平衡：新的激励函数
    • 高效的训练时间：迁移学习

  • LSched 可以与任何基于内存块的数据库系统集成。

  • 一个利用查询编码器提供的信息的深度神经网络

执行模型假设

  • 略。

调度触发器

  • 为了在调度开销和收益之间取得平衡，我们设计了 LSched，使其调度预测器仅在线程和/或查询的重大事件上触发（我们将其称为调度事件）。

调度决策

  • 调度预测器旨在在任何调度事件中学习和执行以下三个调度决策：

    1. 哪些操作员从哪个操作符开始执行以及从哪些查询中进行查询；
    2. 从每个执行根开始，正确的串行顺序；
    3. 每个运行查询的适当并行度。

  • 架构

执行根预测器

  • LSCHED 认为它是潜在流水线的根，然后将该根传递给流水线度预测变量，以找出该流水线的正确程度。

流水线度预测器

  • 预测流水线度是与 Decima 的主要区别，其不能同时安排两个或更多流水线的操作符，以便在同一线程上运行。

并行度预测器

  • 为每个查询计算选择1到最大数量的线程数量的可用线程的概率。
